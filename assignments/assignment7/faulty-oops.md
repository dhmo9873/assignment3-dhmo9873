## 2. Technical Analysis

### Fault Cause
> The primary cause of the crash is a **NULL pointer dereference**.

As indicated in the log: `Unable to handle kernel NULL pointer dereference at virtual address 0000000000000000`. The CPU attempted to perform a write operation (`WnR = 1`) to an unmapped memory address (`0x0`), resulting in a Level 1 translation fault.

### Instruction Pointer (PC) Breakdown
The Program Counter (**pc**) tells us exactly where the kernel was executing when the fault occurred:  
`pc : faulty_write+0x8/0x10 [faulty]`

This reveals the crash happened at an offset of **8 bytes** from the start of the `faulty_write` function within the `faulty` module.

### Register Analysis
At the time of the crash, register **x1** was set to `0000000000000000`. The instruction listed in the `Code` section with parentheses indicates the failing operation: `(b900003f)`. On the ARM64 architecture, this represents a `STR` (Store) instruction attempting to write to the address stored in `x1`.

## 3. Locating the Source Code

### Disassembly via `objdump`
To find the specific line of code, the object file was analyzed using the cross-compilation toolchain:  
`aarch64-linux-objdump -S faulty.o`

**Relevant disassembly snippet:**
```text
0000000000000000 <faulty_write>:
   0:	d2800001 	mov	x1, #0x0  
   4:	d2800000 	mov	x0, #0x0  
   8:	b900003f 	str	wzr, [x1]  <-- CRASH AT OFFSET +8
   c:	d65f03c0 	ret
```
**Mapping to C Code:**  
The disassembly at offset +8 corresponds directly to the following line in the faulty_write function within faulty.c:

```text
ssize_t faulty_write (struct file *filp, const char __user *buf, size_t count, loff_t *pos)
{
	/* make a simple fault by dereferencing a NULL pointer */
	*(int *)0 = 0; // This C instruction maps to assembly offset +8
	return 0;
}
```

By comparing the pc offset from the Oops log to the disassembly generated by objdump, we can prove that the STR instruction at +8 was executing the *(int *)0 = 0; assignment.

## 4. Conclusion
This analysis demonstrates how to debug kernel space faults by correlating runtime "Oops" messages with static binary analysis. Using the Instruction Pointer and Register Dump, we successfully identified that the driver crashed due to an intentional NULL pointer assignment in the faulty_write function.
